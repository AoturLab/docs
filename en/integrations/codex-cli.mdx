Codex CLI can use custom model providers through the user-level `config.toml`. For HiLinkup, use the OpenAI-compatible base URL and the Responses wire API.

<Info>
  Codex configuration uses `~/.codex/config.toml` for user-level settings. A named profile loads `~/.codex/config.toml` first, then overlays `~/.codex/<profile>.config.toml`. Provider settings should be kept in user-level config or a profile, not in project `.codex/config.toml`.
</Info>

<Note>
  Examples use the HiLinkup platform model `qwen3.7-plus`, a Chinese model example. Confirm available code models with `/v1/models` and the console.
</Note>

## Choose the Codex wire API

In Codex CLI custom provider config, `wire_api` decides which endpoint Codex calls.

| Scenario | Recommended config | Endpoint |
| --- | --- | --- |
| Current Codex CLI | `wire_api = "responses"` | `POST /v1/responses` |
| Older Codex CLI or models that only support Chat Completions | `wire_api = "chat"` | `POST /v1/chat/completions` |

<Info>
  If you are not sure which protocol your Codex version should use, start with `responses`. If the request fails, switch to `chat` and verify the Chat Completions cURL example first.
</Info>

## Verify the API first

<Tabs>
  <Tab title="macOS / Linux">
    ```bash
    export HILINKUP_API_KEY="YOUR_HILINKUP_API_KEY"

    curl https://hilinkup.com/v1/models \
      -H "Authorization: Bearer $HILINKUP_API_KEY"
    ```
  </Tab>
  <Tab title="Windows PowerShell">
    ```powershell
    $env:HILINKUP_API_KEY = "YOUR_HILINKUP_API_KEY"

    curl.exe https://hilinkup.com/v1/models `
      -H "Authorization: Bearer $env:HILINKUP_API_KEY"
    ```
  </Tab>
</Tabs>

## Option 1: edit user config

Recommended for long-term use.

<Tabs>
  <Tab title="macOS">
    ```bash
    mkdir -p ~/.codex
    nano ~/.codex/config.toml
    ```

    Paste:

    ```toml
    model = "qwen3.7-plus"
    model_provider = "hilinkup"

    [model_providers.hilinkup]
    name = "HiLinkup"
    base_url = "https://hilinkup.com/v1"
    env_key = "HILINKUP_API_KEY"
    wire_api = "responses"
    request_max_retries = 2
    stream_max_retries = 5
    stream_idle_timeout_ms = 300000
    ```

    Save with `Ctrl + O`, press Enter, then exit with `Ctrl + X`.

    Store the key:

    ```bash
    nano ~/.zshrc
    ```

    Add:

    ```bash
    export HILINKUP_API_KEY="YOUR_HILINKUP_API_KEY"
    ```

    Reload:

    ```bash
    source ~/.zshrc
    ```
  </Tab>
  <Tab title="Linux">
    ```bash
    mkdir -p ~/.codex
    nano ~/.codex/config.toml
    ```

    Paste:

    ```toml
    model = "qwen3.7-plus"
    model_provider = "hilinkup"

    [model_providers.hilinkup]
    name = "HiLinkup"
    base_url = "https://hilinkup.com/v1"
    env_key = "HILINKUP_API_KEY"
    wire_api = "responses"
    request_max_retries = 2
    stream_max_retries = 5
    stream_idle_timeout_ms = 300000
    ```

    Save with `Ctrl + O`, press Enter, then exit with `Ctrl + X`.

    Store the key in your shell profile:

    ```bash
    nano ~/.bashrc
    ```

    or:

    ```bash
    nano ~/.zshrc
    ```

    Add:

    ```bash
    export HILINKUP_API_KEY="YOUR_HILINKUP_API_KEY"
    ```
  </Tab>
  <Tab title="Windows PowerShell">
    ```powershell
    New-Item -ItemType Directory -Force "$env:USERPROFILE\.codex"
    notepad "$env:USERPROFILE\.codex\config.toml"
    ```

    Paste and save:

    ```toml
    model = "qwen3.7-plus"
    model_provider = "hilinkup"

    [model_providers.hilinkup]
    name = "HiLinkup"
    base_url = "https://hilinkup.com/v1"
    env_key = "HILINKUP_API_KEY"
    wire_api = "responses"
    request_max_retries = 2
    stream_max_retries = 5
    stream_idle_timeout_ms = 300000
    ```

    Store the API key:

    ```powershell
    [Environment]::SetEnvironmentVariable(
      "HILINKUP_API_KEY",
      "YOUR_HILINKUP_API_KEY",
      "User"
    )
    ```

    Restart PowerShell, or set it for the current window:

    ```powershell
    $env:HILINKUP_API_KEY = "YOUR_HILINKUP_API_KEY"
    ```
  </Tab>
</Tabs>

Run:

```bash
codex
```

## Common issues

<AccordionGroup>
  <Accordion title="Codex still does not use HiLinkup">
    Check that `model_provider = "hilinkup"` is in the active user-level config. When using `--profile hilinkup`, the file must be `~/.codex/hilinkup.config.toml`, or `%USERPROFILE%\.codex\hilinkup.config.toml` on Windows.
  </Accordion>
  <Accordion title="Codex cannot find the API key">
    Confirm `env_key = "HILINKUP_API_KEY"` and verify that the same terminal can read it. Use `echo $HILINKUP_API_KEY` on macOS/Linux or `echo $env:HILINKUP_API_KEY` in PowerShell.
  </Accordion>
  <Accordion title="Responses does not work">
    Prefer `wire_api = "responses"` for current Codex CLI. If the model or older Codex version only supports Chat Completions, change `wire_api` to `"chat"` and verify [Chat Completions](/en/api-reference/chat-completions) first.
  </Accordion>
  <Accordion title="Do not put provider settings in project .codex/config.toml">
    Codex project-level `.codex/config.toml` cannot override `model_provider`, `model_providers`, and provider authentication settings. Put them in user-level `~/.codex/config.toml` or `~/.codex/hilinkup.config.toml`.
  </Accordion>
  <Accordion title="Do not override the built-in openai provider">
    Do not define `[model_providers.openai]`. `openai` is a reserved built-in provider ID. Use a custom ID such as `hilinkup`.
  </Accordion>
</AccordionGroup>
